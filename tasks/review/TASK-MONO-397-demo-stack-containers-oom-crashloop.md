# 실측 (2026-07-14, 데모 호스트 `m6i.2xlarge` / 32 GB / profile=full)

부팅 완주 직후(`[demo] up complete`) 재시작 횟수:

| 컨테이너 | RestartCount | 리밋 | 원인 |
|---|---|---|---|
| `ecommerce-kafka` | **14** | **536,870,912 = 512 MiB** | ✅ **cgroup OOM (증명됨)** |
| `erp-platform-masterdata` | 14 | **없음** | ❓ **미상** — OOM 아님 |
| `erp-platform-approval` | 14 | **없음** | ❓ 미상 |
| `erp-platform-notification` | 12 | **없음** | ❓ 미상 |
| `finance-platform-ledger` | 5 | **없음** | ❓ 미상 |

kafka 에 대해서는 커널 로그가 원인을 못박는다 — **호스트 OOM 이 아니라 cgroup(컨테이너별 리밋) OOM 이다**:

```
oom-kill:constraint=CONSTRAINT_MEMCG, ... task=java
Memory cgroup out of memory: Killed process (java) total-vm:3859948kB, anon-rss:481028kB
```

`anon-rss ≈ 470 MB` 에서 죽었다 — **512 MiB 리밋의 벽이다.** 같은 시각 호스트는 **7.9 GB 여유**였다. 즉 **호스트를 키워도 안 고쳐진다.** 리밋이 틀린 것이다.

> **512 MiB 짜리 Kafka 브로커는 Kafka 브로커가 아니다.**

## 왜 지금까지 안 보였나

- **CI 는 이 compose 를 안 돌린다** — scm/erp/fan e2e 는 Testcontainers 기반이다(`TASK-MONO-353` 이 `bitnami/kafka:3.7` 삭제를 발견했을 때와 **똑같은 사각지대**).
- 로컬 `demo-up.sh` 는 대개 `demo-core`(4개 프로젝트)로 돌린다 — 메모리 압박이 다르다.
- **크래시루프는 초록으로 보인다.** `docker ps` 는 `Up 16 seconds` 를 보여주고, 재시작 정책이 다시 띄우고, 최종적으로 healthcheck 를 통과하기도 한다. **RestartCount 를 세지 않으면 아무 일도 없어 보인다.**
- **그리고 이 클래스는 이미 한 번 물었다** — 바로 아래 elasticsearch 의 주석: *"`TASK-BE-406`: 1G → 2G. … 로드된 CI 러너에서 `ecommerce-elasticsearch is unhealthy` nightly flake"*. **같은 진단, 같은 처방.** 그때 아무도 *"그럼 kafka 는?"* 을 묻지 않았다.

# 로컬 실측 (2026-07-14, `apache/kafka:3.7.0`) — **동일 부하 대조**

부하: 토픽 10 × 파티션 3, 각 2000 × 1KB 프로듀스. **두 경우 모두 적재를 확인**했다(t5 = 2000).

| 리밋 | JVM 힙 (25% ergonomic) | 부하 후 RSS | **사용률** | 여유 |
|---|---|---|---|---|
| **512M** (현재) | 128 MiB | 425.9 MiB | **83.2%** | 86 MiB |
| **1G** (제안) | 256 MiB | 459.4 MiB | **44.9%** | 565 MiB |

## ⚠️ 리밋은 상한이 아니라 **설정**이다

`KAFKA_HEAP_OPTS` 가 없으므로 JVM 이 **컨테이너를 인지해** 힙을 리밋의 25%(`MaxRAMPercentage` 기본값)로 잡는다 — 실측 `MaxHeapSize = 134217728` (= 128 MiB) at 512M.

⇒ **512M 을 주는 것은 *"512MB 까지 써라"* 가 아니라 *"128 MiB 힙으로 브로커를 돌려라"* 다.**

## 🔴 그리고 **512M 은 격리 상태에선 죽지 않는다** — 이게 이 결함의 핵심이다

혼자 띄운 512M kafka 는 **200 MB 프로듀스를 버틴다**(77~83%, RestartCount 0, healthcheck 통과). `docker ps` 에 `healthy` 로 보인다.

**죽는 건 함대가 붙을 때다.** 데모 호스트에서 12개 ecommerce 서비스가 컨슈머로 연결되자 남은 17% 를 커넥션·페치 버퍼가 먹고 OOM 이 났다.

> **⇒ "죽었는가" 를 묻는 가드는 512M 을 통과시켰을 것이다. 물어야 할 것은 "여유가 얼마나 남았는가" 이다.**

### 재현하지 못한 것을 재현했다고 하지 않는다

컨슈머 12그룹을 `docker exec` 로 붙여 91.7% → `CONSTRAINT_MEMCG` OOM → RestartCount 0→1 을 **관측하긴 했다.** 그러나 그 컨슈머 120개 JVM 은 **kafka 와 같은 cgroup 안**에 있었다 — 실제 데모에선 다른 컨테이너다. **오염된 실험이므로 재현으로 치지 않는다.** 가드는 대신 **여유(headroom)를 요구**한다.

# 이 결함이 다른 결함의 원인일 수 있다 — **인과를 확정하지 않는다**

`TASK-MONO-389` 는 다음을 관측했다: 웜업 도중 정지당한 인스턴스가 **이후 모든 부팅에서** kafka 기동에 실패했다.

```
FATAL: The log dir /tmp/kafka-logs/account.status.changed-0 does not have a topic ID,
       which is not allowed when running in KRaft mode.
```

389 는 이를 **인스턴스 정지가 웜업을 잘라 로그 디렉터리를 반쯤 쓴 채 남긴 것**으로 읽었고, 그 근거로 처녀 인스턴스에서 kafka 가 `healthy` 가 된 것을 들었다.

**그러나 OOM-킬 역시 같은 손상을 만들 수 있다** — 파티션 디렉터리를 만든 뒤 메타데이터를 커밋하기 전에 JVM 이 죽으면 똑같이 topic ID 없는 log dir 이 남는다. 그리고 그 OOM-킬은 **처녀 인스턴스에서도 14회 일어났다.**

⇒ **둘 중 무엇이 그 손상을 만들었는지 389 는 증명하지 않았고, 이 티켓도 아직 증명하지 않는다.** 리밋을 고친 뒤 **재현을 시도해서** 가른다. (389 의 유휴가드 수정은 그 자체로 독립적으로 증명됐다 — 배포된 Lambda 가 `idle_sec = 342`(= 가동 초)를 반환한다. 그 명제는 이 티켓과 무관하게 참이다.)

# 설계

## D1 — kafka 리밋 **512M → 1G**

실측 표대로. `KAFKA_HEAP_OPTS` 는 **넣지 않는다** — 힙을 리밋의 25% 로 따라오게 두면 숫자가 하나로 유지된다. 대신 그 **결합 관계를 주석에 명시**한다(리밋을 만지는 사람이 힙도 함께 만진다는 사실을 모르는 것이 이 결함의 절반이었다).

## D2 — 가드 (u): **"죽었는가" 가 아니라 "여유가 얼마나 남았는가"**

`verify-demo-wrapper.sh --live` 에 추가. **compose 에서 image·env·limit 을 읽어** 그대로 띄우고, 고정 부하를 걸고, **사용률 ≤ 75%** 를 단언한다.

- **열거하지 않는다** — 리밋을 어떤 값으로 바꾸든 가드는 **그 값으로** 잰다.
- **"살아 있음" 을 합격 조건으로 쓰지 않는다** — 512M kafka 는 격리 상태에서 **살아 있고 healthcheck 도 통과한다.** 그런 가드는 이 결함을 **통과시켰을 것이고, 실제로 몇 달간 그랬다.**
- **부하가 들어갔는지 먼저 확인한다**(`t5` 오프셋 ≥ 2000). 무부하 상태의 낮은 사용률로 초록을 내면 **아무것도 보지 않은 것**이다.
- 임계 75%: 512M(83.2%) 와 1G(44.9%) 사이에서 넉넉히 갈린다. 부하가 **프로듀서 전용**이라 함대의 클라이언트 40여 개를 충실히 모사하지 못하므로, **나머지 25% 는 그 모사하지 못한 간극의 몫**이다.

## D3 — **다른 프로젝트에 리밋을 새로 추가하지 않는다**

7개 프로젝트가 리밋 0개다. 여기에 리밋을 **새로 넣는 것은 새 결함을 만드는 일**이다 — 지금 무제한인 JVM 들이 갑자기 25% 힙으로 재설정된다. 그 판단은 **총 예산 실측 후 별도 티켓**이다(§ 미해결).

# 🔴 가드의 첫 판본은 **CI 에서 장식**이었다 — 측정 전용 PR 이 그걸 잡았다

첫 판본의 술어는 **RSS 사용률 ≤ 75%** 였다. 내 노트북에서 mutation 을 돌려 **512M → 83.84% RED / 1G → 46.51% GREEN** 을 확인하고 *"검증 완료"* 라고 적었다.

그런데 impl PR 의 CI 로그가 **1G 에서 31.20%** 를 찍었다 — 내 호스트는 46.51% 였다. **같은 리밋인데 숫자가 다르다.** 그렇다면 `83.84% > 75%` 는 **내 노트북에 대한 명제**이지 CI 에 대한 명제가 아니다.

`MONO-360` 의 규칙대로 **측정 전용 PR(#2533)** 로 물었다 — 512M 으로 되돌린 것 외엔 아무 변경 없음:

| 호스트 | 512M (결함) | 1G (수정) |
|---|---|---|
| 내 Windows (WSL2) | RSS 425.9 MiB = **83.2%** → RED | 459.4 MiB = 44.9% |
| **CI 러너 (ubuntu)** | RSS 193.2 MiB = **38.2%** → **✅ 통과** | 317.3 MiB = 31.2% |

**같은 컨테이너·같은 리밋·같은 부하인데 RSS 가 2.2배 다르다**(WSL2 의 `docker stats` 회계 차이). **CI 러너에서는 512M(38%)과 1G(31%)의 분리가 사실상 없다** — 어떤 임계로도 결함을 가려낼 수 없다.

> **내가 임계를 보정한 83% 는 신호가 아니라 아티팩트였다. 그 위에 세운 가드는 CI 에서 아무것도 지키지 않는다.**
> **측정 전용 PR 을 안 돌렸다면 그대로 머지됐고, 나는 "가드를 넣었다" 고 보고했을 것이다.**

## 고친 술어 — **JVM 에게 자기 힙을 직접 묻는다** (호스트 무관)

```
512M → MaxHeapSize = 134217728 (128 MiB)   ← 128 MiB 힙으로 도는 Kafka 브로커
1G   → MaxHeapSize = 268435456 (256 MiB)
```

`MaxRAMPercentage`(25%)는 **JVM 레벨 규칙**이라 호스트와 무관하다 — 로컬 실측에서 두 값 모두 기대치와 **바이트 단위 일치**했다. 그리고 가드는 그 25% 규칙을 **가정하지 않고 JVM 에게 묻는다**: 누가 `KAFKA_HEAP_OPTS` 로 결합을 끊어도 여전히 진실을 잰다.

**RSS 는 관측으로만 찍고 단언하지 않는다.** 호스트마다 2배씩 달라지는 값 위에 단언을 세우지 않는다.

|---|---|

| **M1 (512M)** | **RED** — 힙 128 MiB < 256. **CI 러너에서 재확인**(#2533 재사용) — **무는지는 그것이 도는 곳에서 증명한다** |
| **M2 (리밋/힙을 못 읽음)** | **공허 FAIL**. 첫 판본이 렌더 들여쓰기를 12칸으로 하드코딩했을 때(compose 는 10칸) 이 분기가 **실제로 발화**했다 |
| **M3 (부하 미적재)** | 코드 경로는 있으나 **실행 검증 안 함.** 그렇게 적는다 |
- [ ] CI GREEN.

3. **kafka 로그 손상의 인과** (AC-4) — `MONO-389` 는 유휴정지를, 이 티켓은 OOM 을 후보로 든다. 둘 다 증명되지 않았다.

# Edge Cases

- **리밋은 상한이 아니라 설정이다.** JVM 이 컨테이너를 인지해 힙을 리밋의 25% 로 잡는다. **리밋을 만지면 힙도 만지는 것이다.** 이걸 모르면 "512MB 면 넉넉하지" 라는 판단이 나온다 — 실제로는 "128 MiB 힙으로 카프카를 돌려라" 였다.
- **리밋을 없애면(무제한) 호스트 OOM 으로 옮겨간다** — 32 GB 에 96개 컨테이너다. 무제한은 답이 아니다.
- **`docker ps` 는 크래시루프를 숨긴다.** `Up 16 seconds` 는 건강해 보인다.
- **`OOMKilled=false, ExitCode=0` 으로 보고된다** — cgroup 이 JVM *스레드*를 죽였고 컨테이너 자신은 깨끗이 종료했기 때문이다. **플래그만 보면 OOM 을 놓친다. 커널 `dmesg` 의 `CONSTRAINT_MEMCG` 가 권위다.**

# Failure Scenarios

- **F1 — 호스트 타입만 올린다** → cgroup 리밋은 그대로다. **아무것도 안 고쳐지고 비용만 오른다.**
- **F2 — 리밋을 눈대중으로 올린다** → AC-2 가 실측을 요구하는 이유다.
- **F3 — 가드를 "죽었는가" 로 만든다** → ✅ **실제로 그럴 뻔했다.** 512M kafka 는 격리 상태에서 **안 죽는다.** 그런 가드는 이 결함을 통과시킨다. **가드의 술어를 결함의 실패 모드에 맞춰라.**
- **F4 — 가드가 부하 없이 잰다** → 무부하 사용률은 낮다. **아무것도 안 보고 초록을 내는 가드**가 된다. (u) 는 t5 오프셋으로 적재를 먼저 확인한다.
- **F5 — 다른 프로젝트에도 리밋을 "일관성 있게" 추가한다** → 무제한이던 JVM 들이 갑자기 25% 힙으로 재설정된다. **일관성이 새 결함을 만든다.**
