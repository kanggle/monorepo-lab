# Goal

`infra/demo/verify-demo-wrapper.sh` 의 가드 **(u)** ("kafka 의 메모리 리밋이 브로커를 돌릴 만한가")는 **단일 프로젝트를 하드코딩한다**:

```sh
u_render="$(render ecommerce)"
```

⇒ 저장소의 kafka 브로커 **6개 중 5개**(wms · iam · scm · fan · erp)와, `TASK-FIN-BE-059` 가 추가할 **일곱 번째**(finance)가 이 가드의 **사정거리 밖**이다. *Demo wrapper smoke* 가 초록이어도 그 6개에 대해서는 **아무것도 재지 않았다.**

이 task 의 목표는 가드를 기계적으로 확장하는 것이 **아니다.** 순진한 확장은 즉시 5개를 RED 로 만들고, 그것을 고치려면 이미 반대로 결정된 일을 해야 한다. 목표는 **"이 가드의 올바른 모집단은 무엇인가"를 판정하고, 그 판정을 코드로 굳히는 것**이다.

## 🔴 왜 "전부 열거" 가 오답인가 — D3 와 정면충돌

`TASK-MONO-397` 의 **D3** 와 `TASK-MONO-399` 의 Out-of-scope 가 이미 결정했다:

> **리밋은 상한이 아니라 *설정*이다** — `KAFKA_HEAP_OPTS`/`JAVA_OPTS` 가 없는 JVM 은 cgroup 리밋의 **25%** 를 힙으로 잡으므로, 리밋을 넣는 순간 지금 무제한인 JVM 90개가 전부 25% 힙으로 **재설정된다**. **일관성이 새 결함을 만든다.**

⇒ 가드 (u) 를 7개 compose 로 확장하면 5개가 `리밋을 못 읽었습니다 — 가드가 공허합니다` 로 즉시 FAIL 하고, 그 FAIL 을 없애는 유일한 방법은 **D3 가 증거 없이 하지 말라고 못 박은 바로 그 일**(리밋 신규 추가)이다. **가드를 고치려다 가드가 막으려던 결함을 만드는 형태다.**

3. `TASK-FIN-BE-059` 가 추가할 finance 브로커가 선택된 술어 하에서 어떻게 취급되는지 명시한다.

# Acceptance Criteria

**AC-0 — 착수 전 재측정 (verify-then-act).** 위 표(6 브로커 / 리밋 1건)를 `origin/main` 에서 **다시 센다.** 인계된 숫자는 출처가 아니라 **가설**이다. `TASK-FIN-BE-059` 가 먼저 머지됐다면 브로커는 7개다. 다르면 그 사실이 먼저다.

**AC-1 — 모집단을 판정한다.** (A)/(B)/(C) 중 하나를 **근거와 함께** 선택한다. (C)를 고르려면 D3 를 뒤집는 **실측 근거**를 제시해야 한다 — 없으면 (C)는 선택 불가이며, 그렇게 적는다.

**AC-2 — 가드가 공허하지 않음을 증명한다.** 선택한 술어가 **실제로 무는지** 픽스처로 보인다: 위반 케이스(예: (B)라면 리밋 512M 선언)를 넣으면 RED, 원복하면 GREEN. **"통과했다"는 물었다는 증거가 아니다** — 가드 (u) 자신이 첫 판본에서 들여쓰기 하드코딩으로 리밋을 못 읽고 공허 FAIL 을 냈던 전례가 그 파일에 적혀 있다.

**AC-3 — 커버리지를 출력한다.** 검사된 브로커 수 / 전체 브로커 수를 로그로 찍는다((h) 가 `이미지 실재 검증 커버리지 N/M` 로 이미 하는 것과 같은 관행). 어떤 술어를 고르든 **검사되지 않은 브로커가 몇 개인지 사람이 읽을 수 있어야 한다.**

**AC-4 — 오탐 0.** 현재 `main` 의 6(또는 7) 브로커 전부에 대해 선택한 술어가 **의도한 판정**을 낸다. D3 를 존중하는 (A)/(B) 하에서 미선언 브로커는 통과해야 하며, 통과가 **묵인이 아니라 결정**임이 로그에 드러나야 한다.

**AC-5 — 문서 정합.** 판정을 `infra/demo/README.md`(또는 가드 주석)에 반영한다. `TASK-MONO-399` 가 AC-2 에서 실제 메모리 예산을 재고 D3 를 재검토할 수 있으므로, **399 가 이 판정을 뒤집을 수 있는 후속임을 상호 참조**한다.

# Edge Cases

- **`render` 는 compose 렌더 결과다 — 원본 YAML 이 아니다.** 들여쓰기·기본값이 원본과 다르다. 가드 (u) 의 첫 판본이 `memory:` 를 12칸으로 가정했는데 compose 는 10칸으로 렌더해서 리밋을 못 읽었다. **칸수를 하드코딩하지 않는다**(현재 판본의 awk 는 이미 이 교훈을 반영).
- **`kafka` 라는 서비스 키가 7개 프로젝트에 중복된다.** 열거로 확장한다면 프로젝트별로 `render <project>` 를 돌려야 하며, 한 번의 전역 grep 은 어느 브로커의 리밋인지 구분하지 못한다.
- **`memory:` 미선언과 `memory: 0` 은 다른 명제다.** 전자는 "무제한", 후자는 명시적 0. 술어가 둘을 같게 취급하면 안 된다.
- **finance 브로커는 이 task 착수 시점에 존재할 수도, 안 할 수도 있다.** `TASK-FIN-BE-059` 와 순서 의존이 없도록 술어는 **발견된 브로커 집합**에 대해 동작해야 한다(하드코딩 목록 금지 — 그것이 애초의 결함이다).
- **`docker` 가 필요한 가드다.** (u) 는 실기동까지 한다. 열거로 확장하면 기동 횟수가 브로커 수만큼 늘어 CI 시간이 선형 증가한다. **정적 판정과 실기동을 분리**할지 결정해야 한다(실기동은 대표 1개로 충분할 수 있다).

# Failure Scenarios

- **가장 위험: 가드를 확장하려다 D3 를 조용히 뒤집는 것.** "일관성" 을 이유로 5개 compose 에 리밋을 새로 넣으면, 리밋 없는 JVM 들이 전부 cgroup 25% 힙으로 **재설정**된다. 397 이 명시적으로 결정해 둔 것을 근거 없이 되돌리는 형태이며, **가드가 막으려던 종류의 결함을 가드 수정이 만든다.**
- **공허한 확장.** 열거는 했는데 파싱이 브로커별로 안 맞아 리밋을 못 읽고, 그럼에도 통과하는 판본. AC-2 의 픽스처가 이것을 막는다. **초록은 물었다는 증거가 아니다.**
- **커버리지를 말하지 않는 것.** (A)를 골라도 정당하다 — 단, 6개가 검사되지 않는다는 사실이 **로그에 없으면** 다음 사람은 다시 "가드가 있으니 괜찮다"고 읽는다. 이 task 가 존재하는 이유가 정확히 그것이다.
- **동적 측정을 술어로 승격시키는 것.** 397 이 두 번 실패했다. 호스트 의존값은 문서를 고치는 데 쓰고 술어로 쓰지 않는다.
