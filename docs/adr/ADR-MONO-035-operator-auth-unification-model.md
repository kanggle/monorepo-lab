## 1. Context

### 1.1 Where ADR-032 step 4 stands, and the gap it exposes

ADR-032 D5 step 4 is "drop legacy `account_type` (claim, column, dual-read gateway leg; SSO cross-type lifted)." Steps 0-3 made the `roles` claim live and additive: consumer tokens now carry `CUSTOMER`/`FAN` (via stored `account_roles` or the `RoleSeedPolicy` aud-default seed, ADR-033), and both domain gateways dual-read (`hasRole(...) OR account_type`). For **consumers**, the `roles` leg fully admits: the `ecommerce` client is `tenant_id='ecommerce'`, so `RoleSeedPolicy.seed('ecommerce','CONSUMER')=["CUSTOMER"]`, and the ecommerce gateway's consumer path admits on `CUSTOMER`. The consumer half of step 4 is unblocked.

The **operator** half is not. The investigation at `9d4e878c3` found operators carry **no `roles` claim**:

1. **No seed.** Operators base-login through the `platform-console-web` OIDC client, which is `tenant_id='gap'` (V0015 ŌĆö "the platform's own operational tenant slug"). `RoleSeedPolicy.seed('gap', 'OPERATOR')` hits the `default` branch ŌåÆ `[]`. The seed table covers `ecommerce`/`wms`/`scm`/`erp`/`mes`/`fan-platform`, never `gap`.
2. **No stored roles.** No `account_roles` rows exist for any operator (repo-wide `INSERT INTO account_roles` = one migration-rebuild test). The federation/console seeds set `credentials.account_type='OPERATOR'` and seed `admin_operators` + `operator_tenant_assignment`, but never `account_roles`.
3. **Nothing to preserve.** The assume-tenant exchange (BE-370) does `getClaimAsStringList("roles")` on the subject (base) token and preserves it verbatim. The base operator token has no `roles`, so the assumed token has none either.

Therefore an operator's base **and** assumed tokens carry no `roles`, and every domain gateway admits them **only** via the legacy `account_type=OPERATOR` leg (`AccountTypeEnforcementFilter`: `hasRole(roles,"ECOMMERCE_OPERATOR") || "OPERATOR".equals(accountType)`; `AccountTypeValidationFilter`: `non-empty roles || "OPERATOR".equals(accountType)`). Step 2 (BE-369/370) was **net-positive additive** ŌĆö it gave consumers real roles while leaving the `account_type` leg in place; the dual-read masked that operators never gained a role source. Dropping the `account_type` emission today would lock every operator out of every domain gateway.

### 1.2 Why this is a structural gap, not an oversight

The base operator token is issued for `aud=platform-console` / `tenant_id='gap'`. It has **no domain context** ŌĆö the operator has not yet selected which customer tenant (and therefore which domain) they will act in. Domain roles (`WMS_OPERATOR`, `ECOMMERCE_OPERATOR`, ŌĆ”) are meaningful only **per assumed tenant**: the same operator assigned to a wms-entitled tenant is a `WMS_OPERATOR` there and to an ecommerce-entitled tenant is an `ECOMMERCE_OPERATOR` there. ADR-033 S4's "preserve the operator's roles from the base token" implicitly assumed the base token has a domain-role set to preserve; for operators it structurally cannot, because the base token predates tenant/domain selection. The natural place to source an operator's domain role is therefore the **assume-tenant exchange**, which already (a) validates the operator, (b) calls admin-service `GET /internal/operator-assignments/check` to fail-closed-verify the operator's assignment to the selected tenant, and (c) populates the selected tenant's `entitled_domains` claim. The exchange already knows the operator and the selected tenant's domains ŌĆö everything needed to derive the domain role.

### 1.3 The two operator credentials (what "login/credential consolidation" must converge)

ADR-034 ┬¦ 1.1 found the operator is the genuinely-disjoint identity. The base-state detail relevant to step 4:

| | Consumer-side OIDC credential | Admin-panel credential |
|---|---|---|
| Store | `auth_db.credentials` (`account_type='OPERATOR'`, argon2id `credential_hash`) | `admin_db.admin_operators.password_hash` (argon2id) |
| Login | auth-service SAS form-login (`CredentialAuthenticationProvider`) ŌåÆ GAP OIDC base token (the platform-console login) | admin-service `AdminLoginService` ŌåÆ `OperatorAccessTokenIssuer` (`iss=admin-service`, own RSA JWKS, `token_type=admin`) |
| Purpose | the platform-console **OIDC base login** (then assume-tenant ŌåÆ domain tokens, and admin-service token-exchange ŌåÆ admin RBAC token) | the admin-service **admin-panel RBAC** login (admin-console APIs) |
| 2FA | none | TOTP (`admin_operator_totp`, admin_db; enforced only when `anyRoleRequires2fa`) |

Both passwords are argon2id with **identical parameters** (`libs/java-security` `Argon2idPasswordHasher`, m=65536/t=3/p=1) ŌĆö so a single credential store is mechanically feasible. The barrier is structural: two passwords for (potentially) one person, in two databases, bridged only by `admin_operators.oidc_subject` (and now `identity_id`, ADR-034 3c). There are **two orthogonal, both-active** token-exchange paths: admin-service `POST /api/admin/auth/token-exchange` (OIDC base token ŌåÆ admin RBAC `token_type=admin` token, ADR-014) and auth-service assume-tenant RFC 8693 (OIDC base token ŌåÆ tenant-scoped domain token, ADR-020). "Operator credential consolidation" means folding the admin-panel password store into the OIDC credential while preserving login availability and TOTP.

### 1.4 The underspecified points this ADR must resolve (HARDSTOP-09)

- **(a)** What is the operator's JWT-domain-role **source**, given the seed returns `[]` for `gap`, `account_roles` is empty for operators, and `admin_operator_roles` is disjoint (ADR-033 S2)? ŌĆö **O1**.
- **(b)** How do the two operator credentials **converge** without dragging the operator login/TOTP security path into a risky big-bang? ŌĆö **O2**.
- **(c)** How is the `credentials.identity_id` cross-store reference (ADR-034 U6 3b, deferred to step 4 because it is login-path work) added? ŌĆö **O3**.
- **(d)** How are TOTP/2FA + operator-specific attributes (`oidc_subject`, `finance_default_account_id`) handled under convergence? ŌĆö **O4**.
- **(e)** How does the whole of step 4 (operator roles ŌåÆ `account_type` drop ŌåÆ credential convergence ŌåÆ `identity_id`) stage additively with **zero mis-authorization window**? ŌĆö **O5**.

Each would silently bake the operator-auth model if chosen in code. The `account_type` claim removal **does** touch `jwt-standard-claims.md` (unlike ADR-033/034): the contract was rewritten at step 0 (MONO-255) to mark `account_type` Deprecated/migration-only with removal at step 4, and step 4b **finalizes** that removal per the contract's ┬¦ Change Rule. This ADR records the decision (HARDSTOP-09 remediation: decide first, PAUSE until ACCEPTED); the contract finalization + implementation are post-ACCEPTED execution tasks.

## 2. Decision

> Direction is **CHOSEN-PROPOSED**; finalised (byte-unchanged) at ACCEPTED. Each decision lists the chosen option + the rejected alternatives. **No code/contract/schema change in this ADR.** O1/O2 were surfaced from the base-state investigation (operators ride the `account_type` leg; two argon2id credentials in two DBs) at `9d4e878c3`.

### O1 ŌĆö operator JWT-domain-role source (the crux that unblocks the `account_type` drop): derive at assume-tenant from the selected tenant's entitled domains

| Option | Mechanics | Verdict |
|---|---|---|
| **A. Derive the operator's domain role(s) at ASSUME-TENANT issuance time from the SELECTED tenant's domains. The exchange already validates the operator + fail-closed-checks their assignment (`OperatorAssignmentCheckUseCase`) + resolves the selected tenant's `entitled_domains`. Add a server-side derivation: for each domain the selected tenant is entitled to (and the operator is assigned to that tenant), the assumed token's `roles` includes that domain's operator role (`wms ŌåÆ WMS_OPERATOR`, `ecommerce ŌåÆ ECOMMERCE_OPERATOR`, `scm ŌåÆ SCM_OPERATOR`, `erp ŌåÆ ERP_OPERATOR`, `fan-platform ŌåÆ FAN_OPERATOR`, ŌĆ”) ŌĆö the operator-role mirror of the consumer `RoleSeedPolicy` aud-default seed, keyed on the ASSUMED tenant's entitled domains instead of the login client's platform. This REPLACES BE-370's preserve-from-base for the operator case (the base token has no domain context to carry). The base operator token (aud=platform-console) needs no domain roles ŌĆö only assumed (aud=domain) tokens hit the domain gateways.** | The assumed token carries `roles=[WMS_OPERATOR]` (or the union across the selected tenant's entitled domains), so the domain gateway's `roles` leg admits the operator on its own ŌĆö and the `account_type` leg can be dropped (O5 4b). Reuses the existing fail-closed assignment-check + `entitled_domains` machinery (no new admin call). Admin-console RBAC (`admin_operator_roles`) is **untouched** ŌĆö the domain role is derived from the assignment+tenant-domain, never copied from admin roles (O6 / ADR-033/034-U5 preserved). | **CHOSEN** ŌĆö sources the role exactly where domain context first exists (the assumed token); the assignment-check already gates it fail-closed; coarse-but-correct for gateway admission (an operator assigned to act in a tenant gets that tenant's domain operator role); keeps the two role namespaces disjoint; the smallest change that unblocks the `account_type` drop. The wms gateway needs only a non-empty `roles`; ecommerce admin needs `ECOMMERCE_OPERATOR` specifically ŌĆö both satisfied by the entitled-domain-derived role. |
| B. Per-assignment explicit domain roles on `operator_tenant_assignment` (extend `permission_set_id`/add a domain-role column; provisioning sets them). | Fine-grained: a different role per (operator, tenant). | **Rejected for step 4 (recorded as a deferred refinement)** ŌĆö heavier (new grant data + a provisioning/admin UX to populate it) and `permission_set_id` is currently always-NULL/unused; per-assignment granularity is not required for gateway admission (which only needs a domain operator role present). A's entitled-domain derivation gives correct admission today; per-assignment roles can layer on later without changing the model. |
| C. Map `admin_operator_roles` (admin-console RBAC) ŌåÆ JWT domain `roles`. | Reuse the operator's existing admin grants. | **Rejected** ŌĆö violates ADR-033 S2 / ADR-034 U5 disjointness: `admin_operator_roles` are admin-console RBAC (`SUPER_ADMIN`, `SUPPORT_*`, `TENANT_ADMIN`), **not** domain roles (`SUPER_ADMIN` is not `WMS_OPERATOR`); folding them into the JWT `roles` claim is exactly the namespace merge ADR-033/034 forbid. |

> **O1 sub-decision ŌĆö fail-soft vs fail-closed.** The domain-role derivation is **additive within the already-fail-CLOSED assignment gate**: the exchange first fail-closed-verifies the assignment (no assignment ŌåÆ no token, unchanged); given an admitted assignment, the role derivation reads the selected tenant's `entitled_domains` (already fail-soft per ADR-019) and emits the derived roles. An empty derivation (e.g. a tenant with no entitled domains) ŌåÆ empty `roles` ŌåÆ the domain gateway 403s (net-zero, the correct least-privilege outcome ŌĆö same fail-soft-population / authz-at-the-gateway split as ADR-033 S5). The assignment gate's fail-CLOSED property is unchanged.

### O3 ŌĆö `credentials.identity_id` (the ADR-034 U6 step-3b leg deferred to step 4)

Add `identity_id` to `auth_db.credentials` (value-convention cross-DB reference, mirroring `admin_operators.identity_id` from ADR-034 3c), backfilled from `accounts.identity_id` via the shared `account_id`. ADR-034 U6 3b explicitly **re-sequenced** the `credentials.identity_id` leg to step 4 because it is login-path work (the credential row is the login anchor). Adding it here completes the central-identity correlation across all three stores (`accounts` 3a, `admin_operators` 3c, `credentials` 4). Additive, net-zero (no caller depends on it until a later consolidation re-keys on it).

### O4 ŌĆö TOTP/2FA + operator-specific attributes under convergence

- **TOTP/2FA stays admin-service-internal** (`admin_operator_totp`, admin_db; enforced when `anyRoleRequires2fa`). Operator 2FA is an admin-panel concern, not folded into the OIDC base login in step 4 (a future step-up-auth on the OIDC side is a deferred follow-up, ADR-032 D4-B). The OIDC base login is unchanged.
- **`oidc_subject`** remains the assume-tenant/admin-token-exchange bridge (the existing seam); `identity_id` (ADR-034 3c + O3) is the durable identity link. **`finance_default_account_id`** stays on `admin_operators` (operator-specific attribute, ADR-034 U7). Neither moves to `credentials`/`accounts`.

## 4. Alternatives Considered

- **Map `admin_operator_roles` ŌåÆ JWT domain roles (O1-C).** Rejected ŌĆö violates ADR-033 S2 / ADR-034 U5 role-namespace disjointness (admin RBAC ŌēĀ domain roles).
- **Per-assignment explicit operator domain roles (O1-B).** Rejected for step 4 (deferred refinement) ŌĆö heavier; entitled-domain derivation suffices for gateway admission.
- **Keep two operator passwords permanently (O2-B).** Rejected ŌĆö never achieves "one account = one credential" for operators.
- **Big-bang operator login + `password_hash` removal + TOTP merge (O2-C).** Rejected ŌĆö drags the auth/TOTP security path into one un-bisectable change with no break-glass margin.
- **Drop `account_type` before operators have a role source (do step 4b first).** Rejected ŌĆö the mis-authorization window this whole ADR exists to avoid (the O5 ordering invariant: 4a precedes 4b).
- **Seed `account_roles` rows for operators (give operators consumer-store roles).** Rejected as the model ŌĆö operator domain capability is per-assumed-tenant (resolved at exchange time), not a static base-login grant; storing per-(operator,tenant) domain roles is the deferred O1-B refinement, not required for admission. (The derivation at assume-tenant is the cleaner source.)

## 8. Amendment ŌĆö operator-create requires an existing tenant account (TASK-MONO-334, 2026-07-07)

**Decision (user-directed, 2026-07-07).** `POST /api/admin/operators` now **requires** the target `email` to already own a signed-up account in the target `tenantId`. The producer verifies existence (account-service `GET /internal/accounts?email&tenantId`) before creating; a missing account is rejected with **`422 OPERATOR_ACCOUNT_NOT_FOUND`**, and account-service unavailability is **fail-closed** (`503 DOWNSTREAM_ERROR` ŌĆö the operator is not created). The console pre-gates submit on the same check (supersedes the TASK-PC-FE-179 fail-soft advisory).

**What this amends.** O6 held "operator login availability preserved via break-glass `password_hash`." That invariant **stands** ŌĆö `admin_operators.password_hash` remains a nullable break-glass credential and is not removed. What changes is the **creation precondition**: an operator can no longer be *born* against an email that has no account, **even with a break-glass password**. Break-glass is now strictly a *secondary* login for an operator whose account already exists (the account's unified IAM/OIDC credential being the PRIMARY login) ŌĆö it is no longer a way to mint an account-less "dangling" operator. This tightens toward the ADR-032/034/036 unified-identity north star (every operator maps to a real Global Account) by closing the dangling-operator escape hatch that TASK-PC-FE-179 only *warned* about.

**Rationale for the trade-off.** Requiring account existence couples operator-create availability to account-service (fail-closed on downstream failure). This is accepted deliberately: the amendment's whole point is "never create an unverified operator," so a verify-then-create gate must fail closed. The **`tenantId='*'` platform-scope bootstrap is EXEMPT** ŌĆö there is no `account_db` row for `*` to verify against, so the SUPER_ADMIN bootstrap path is unchanged. The `FirstAdminProvisioner` self-service onboarding path (ADR-044) does **not** route through `CreateOperatorUseCase` and is therefore unaffected (its caller already owns the account it provisions from).
