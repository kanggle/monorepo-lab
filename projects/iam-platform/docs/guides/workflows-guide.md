# 워크플로우 가이드

프로젝트에서 사용하는 작업 프로세스 문서 목록과 흐름 요약.

## 워크플로우란?

`.claude/workflows/`에 정의된 프로세스 참조 문서. Claude가 자동 실행하는 파이프라인이 아니라, 에이전트가 작업 시 참고하는 매뉴얼.

---

## 워크플로우 목록

### task-implementation (태스크 구현)

일반적인 태스크 구현의 전체 흐름.

```
태스크 읽기/검증 → 스펙 읽기 → 스킬 읽기 → 기존 코드 파악 → 구현 → 테스트 → 셀프 리뷰 → 태스크 이동
```

#### 단계별 상세

| 단계 | 작업 내용 |
|---|---|
| 1. CLAUDE.md 읽기 | 핵심 원칙, 우선순위, Hard Stop 규칙, 필수 워크플로우 파악 |
| 2. 태스크 읽기/검증 | `tasks/ready/<task>.md` 확인, 필수 섹션 존재 여부 검증, 관련 스펙/계약 존재 확인 |
| 3. 스펙 읽기 | `entrypoint.md` 순서에 따라 platform → contracts → services → features → use-cases 순으로 읽기 |
| 4. 스킬 읽기 | `.claude/skills/INDEX.md`에서 태스크 유형에 맞는 스킬 파일만 선택적으로 읽기 |
| 5. 설계 맥락 참조 | 설계 판단이 필요한 경우에만 `knowledge/` 참조 (source of truth 아님) |
| 6. 기존 코드 파악 | 대상 서비스의 코드 패턴, 컨벤션, 구조 파악 후 일관성 유지 |
| 7. 구현 | 스펙/계약에 정의된 범위 내에서 구현, 레이어 의존성 규칙 준수, 계약 필드명 정확히 일치 |
| 8. 테스트 작성 | `testing-strategy.md`에 따른 테스트 작성, 인수 기준 전항목 커버, 엣지 케이스/실패 시나리오 포함 |
| 9. 셀프 리뷰 | 레이어 위반 없음, 계약 필드명 일치, 테스트 통과, 인수 기준 충족, 스펙 범위 초과 없음 확인 |
| 10. 태스크 이동 | `tasks/ready/` → `tasks/in-progress/` → 완료 후 `tasks/review/` |

#### 주요 규칙

- 태스크가 `tasks/ready/`에 없으면 구현 불가
- 필수 섹션(Goal, Scope, Acceptance Criteria, Related Specs, Related Contracts, Edge Cases, Failure Scenarios) 중 하나라도 없으면 Hard Stop
- 스펙과 계약이 존재하지 않으면 구현 전에 먼저 생성해야 함
- 구현 범위는 반드시 스펙에 정의된 범위 내로 제한

#### 사용 에이전트

| 단계 | 에이전트 |
|---|---|
| 아키텍처 결정 | `architect` |
| 백엔드 구현 | `backend-engineer` |
| 프론트엔드 구현 | `frontend-engineer` |
| 테스트 작성/검증 | `qa-engineer` |
| 코드 리뷰 | `code-reviewer` |
| 계약 변경 필요 시 | `api-designer`, `event-architect` |

#### 판단 기준

- 태스크에 계약 변경이 필요한 경우: 먼저 `contract-change` 워크플로우 수행 후 진행
- 태스크에 스펙이 불명확한 경우: Hard Stop 후 스펙 명확화 요청
- `knowledge/` 참조는 설계 판단 근거가 필요할 때만 사용하며, 스펙과 충돌 시 스펙 우선

### contract-change (계약 변경)

API 또는 이벤트 계약을 변경할 때.

```
변경 필요성 확인 → 영향 분석 (breaking change 여부) → 계약 업데이트 → 관련 스펙 동기화 → 구현 반영
```

> 원칙: 계약 변경은 구현보다 먼저 수행한다.

#### 단계별 상세

| 단계 | 작업 내용 |
|---|---|
| 1. 변경 필요성 확인 | 태스크 또는 스펙에서 계약 변경 근거 파악, 기존 계약 내용 확인 |
| 2. 영향 분석 | 해당 계약을 사용하는 모든 서비스 식별, breaking change 여부 판정 |
| 3. 계약 업데이트 | `specs/contracts/` 파일 수정, 버전 정보 갱신, 변경 이유 기록 |
| 4. 관련 스펙 동기화 | 해당 계약을 참조하는 서비스 스펙 확인 및 불일치 발생 시 업데이트 |
| 5. 구현 반영 | 계약 확정 후에만 코드 구현 진행, Request/Response 필드명이 계약과 정확히 일치하는지 검증 |

#### 주요 규칙

- 계약 변경은 반드시 구현 전에 완료해야 함
- Breaking change 판정 기준:
  - 필드 삭제/이름 변경 = breaking
  - 선택 필드 추가 = non-breaking
  - 타입 변경 = breaking
- Breaking change인 경우 영향받는 모든 서비스를 식별하고 대응 계획 수립 필요

#### 사용 에이전트

| 역할 | 에이전트 |
|---|---|
| API 계약 변경 | `api-designer` |
| 이벤트 계약 변경 | `event-architect` |
| 영향 분석 | `architect` |

#### 판단 기준

- 필드 추가(선택)만 있는 경우: non-breaking으로 처리, 하위 호환 유지
- 필드 삭제/이름 변경/타입 변경인 경우: breaking change로 처리, 버전 업 및 마이그레이션 계획 필요
- 계약 변경 후 관련 서비스 스펙에 불일치가 없는지 반드시 확인 후 구현 진행

### event-change (이벤트 변경)

도메인 이벤트를 추가하거나 변경할 때.

```
요구사항 파악 → 이벤트 계약 설계 → 메시징 패턴 결정 → 발행자 구현 → 소비자 구현 → 테스트
```

#### 단계별 상세

| 단계 | 작업 내용 |
|---|---|
| 1. 이벤트 요구사항 파악 | `specs/features/` 또는 `specs/use-cases/`에서 이벤트 시나리오 파악, 발행자/소비자 서비스 식별 |
| 2. 이벤트 계약 설계 | `specs/contracts/`에 이벤트 스키마 정의, 명명 규칙 준수(`{Aggregate}.{과거형동사}`), 필수 필드 포함 |
| 3. 메시징 패턴 결정 | Outbox 패턴 적용 여부, 멱등성 보장 방식, 재시도/DLQ 정책, 순서 보장 요건 결정 |
| 4. 발행자 구현 | 도메인 이벤트 발행 로직 구현, 트랜잭션 내에서 Outbox에 이벤트 저장 |
| 5. 소비자 구현 | 이벤트 핸들러 구현, 멱등성 처리, 실패 시 재시도/DLQ 처리 |
| 6. 테스트 | 이벤트 발행 테스트, 소비자 멱등성 테스트, 실패 시나리오 테스트 |

#### 주요 규칙

- 이벤트 계약 변경은 구현 전에 완료해야 함 (CLAUDE.md Contract Rule)
- 이벤트 명명 규칙: `{Aggregate}.{PastTenseVerb}` (예: `Order.Created`)
- 이벤트 필수 필드: `eventId`, `eventType`, `occurredAt`, `version`, `payload`
- 메시징 스킬(`.claude/skills/messaging/`)이 아직 작성되지 않은 경우: `platform/event-driven-policy.md`와 `specs/contracts/events/` 계약을 주요 지침으로 사용

#### 사용 에이전트

| 역할 | 에이전트 |
|---|---|
| 이벤트 설계 | `event-architect` |
| 발행자/소비자 구현 | `backend-engineer` |
| 테스트 | `qa-engineer` |

#### 판단 기준

- Outbox 패턴 적용 여부: 트랜잭션 일관성이 필요한 이벤트에 적용
- 멱등성 처리: 소비자는 동일 이벤트를 여러 번 수신해도 결과가 동일해야 함
- DLQ 정책: 재시도 횟수 초과 시 Dead Letter Queue로 이동, 별도 모니터링/처리 필요

### service-bootstrap (서비스 생성)

새 서비스를 처음부터 만들 때.

```
아키텍처 결정 → 서비스 스펙 작성 → 계약 정의 → 프로젝트 구조 생성 → 인프라 구성 → 기본 구현
```

#### 단계별 상세

| 단계 | 작업 내용 |
|---|---|
| 1. 아키텍처 결정 | 서비스 도메인 복잡도에 맞는 아키텍처 패턴 선택, `platform/architecture-decision-rule.md` 준수, ADR 문서화 |
| 2. 서비스 스펙 작성 | `specs/services/<service>/architecture.md` 생성 (선택 아키텍처 패턴, 레이어 구조 및 의존 방향, 기술 스택 포함) |
| 3. 계약 정의 | API 계약 및 이벤트 계약을 `specs/contracts/`에 정의 |
| 4. 프로젝트 구조 생성 | 선택 아키텍처에 맞는 패키지/디렉터리 구조 생성, 빌드 설정(Gradle 모듈/package.json), 설정 파일(application.yml/.env) |
| 5. 인프라 구성 | Dockerfile 작성, Kubernetes 매니페스트(해당 시), CI/CD 파이프라인 추가 |
| 6. 기본 구현 | 헬스 체크 엔드포인트, 공통 설정(보안/에러 처리/로깅), DB 연결 및 초기 마이그레이션 |

#### 주요 규칙

- 새 서비스는 반드시 `specs/services/<service>/architecture.md`에 선언된 아키텍처를 따라야 함
- 아키텍처 결정 전에 ADR(Architecture Decision Record)을 먼저 작성
- 서비스 고유 도메인 로직은 `libs/`로 이동 금지 (`platform/shared-library-policy.md`)
- 서비스 스펙 작성 시 `.claude/templates/service-architecture-template.md` 참조

#### 사용 에이전트

| 역할 | 에이전트 |
|---|---|
| 아키텍처 선택 | `architect` |
| API 계약 | `api-designer` |
| DB 스키마 | `database-designer` |
| 백엔드 구현 | `backend-engineer` |
| 프론트엔드 구현 | `frontend-engineer` |
| 인프라 설정 | `devops-engineer` |

#### 판단 기준

- 아키텍처 패턴 선택: 도메인 복잡도에 따라 `architecture-decision-rule.md`에서 적합한 패턴 선택
- 계약 정의 필요 여부: 외부 서비스와 통신이 있는 경우 반드시 계약 먼저 정의
- 인프라 구성 범위: 프로젝트 인프라 정책에 따라 필요 항목 결정

### production-fix (프로덕션 긴급 수정)

프로덕션 장애 대응.

```
문제 파악 → 근본 원인 식별 → 최소 수정 구현 → 테스트 → 배포 → 후속 조치 (별도 태스크 생성)
```

> 원칙: 최소 범위 변경. 긴급해도 테스트 필수.

#### 단계별 상세

| 단계 | 작업 내용 |
|---|---|
| 1. 문제 파악 | 증상, 영향 범위, 발생 시각 확인, 로그/메트릭/에러 트레이스 분석 |
| 2. 근본 원인 식별 | 관련 코드 경로 추적, 최근 배포 변경 확인(`git log`), 재현 가능한 시나리오 식별 |
| 3. 최소 수정 구현 | 문제 해결에 필요한 최소 코드 변경만 적용, 리팩토링/기능 개선/추가 절대 금지, 기존 패턴/컨벤션 유지 |
| 4. 테스트 | 버그 재현 테스트 먼저 작성(실패 확인) → 수정 후 통과 확인 → 기존 테스트 전부 통과 확인 |
| 5. 배포 | hotfix 브랜치에서 작업, 코드 리뷰 진행, 배포 후 모니터링 |
| 6. 후속 조치 | 근본 원인 해결을 위한 별도 태스크를 `tasks/ready/`에 생성, 인시던트 기록 |

#### 주요 규칙

- 최소 범위 원칙: 문제 해결에 필요한 변경만 수행, 그 외 일체 금지
- 테스트 필수: 긴급 상황에도 테스트 없이 배포 불가
- 근본 원인 후속 처리: 핫픽스 후 반드시 별도 태스크로 근본 원인 해결 작업 등록
- 반드시 hotfix 브랜치에서 작업

#### 사용 에이전트

| 역할 | 에이전트 |
|---|---|
| 근본 원인 분석 | `architect`, `backend-engineer` |
| 수정 구현 | `backend-engineer`, `frontend-engineer` |
| 테스트 | `qa-engineer` |
| 배포 | `devops-engineer` |

#### 판단 기준

- 수정 범위 결정: 문제 재현에 최소한으로 필요한 변경만 포함, 관련 없는 코드 건드리지 않음
- 테스트 우선순위: 버그 재현 테스트 → 수정 후 통과 확인 → 회귀 테스트 순서 준수
- 후속 태스크 생성: 핫픽스가 임시 방편인 경우 반드시 근본 원인 해결 태스크 별도 등록

### spec-change (스펙 변경)

스펙을 추가하거나 변경할 때.

```
변경 범위 파악 → 영향 분석 (하위 문서) → 스펙 수정 → 하위 문서 동기화 → 기존 구현 확인
```

> 원칙: 스펙이 source of truth. 구현과 충돌하면 스펙이 우선.

#### 단계별 상세

| 단계 | 작업 내용 |
|---|---|
| 1. 변경 범위 파악 | 변경 대상 스펙 파일 특정, 변경 이유 명확화(신규 기능/요구사항 변경/오류 수정) |
| 2. 영향 분석 | 스펙 우선순위에 따라 하위 문서 영향 분석 (platform → contracts → services → features → use-cases 순) |
| 3. 스펙 수정 | 관련 스펙 파일 편집, 변경 이력(날짜, 이유) 기록, 상위 스펙과 충돌 없는지 검증 |
| 4. 하위 문서 동기화 | 변경된 스펙을 참조하는 하위 스펙/계약 확인, 불일치 발견 시 업데이트, 영향받는 태스크 보고 또는 업데이트 |
| 5. 기존 구현 확인 | 이미 구현된 코드가 새 스펙과 일치하는지 확인, 불일치 시 수정 태스크 생성 |

#### 주요 규칙

- 스펙이 항상 source of truth: 구현이 스펙과 충돌하면 스펙이 우선
- 상위 스펙 변경 시 파급 효과가 크므로 영향 분석을 철저히 수행
- 스펙 변경 후 하위 문서 동기화 누락 금지
- 이미 구현된 코드와 불일치가 발생한 경우, 코드를 직접 수정하지 말고 별도 수정 태스크 생성

#### 사용 에이전트

| 역할 | 에이전트 |
|---|---|
| 아키텍처 스펙 | `architect` |
| API 계약 스펙 | `api-designer` |
| 이벤트 계약 스펙 | `event-architect` |
| DB 스펙 | `database-designer` |

#### 판단 기준

- 변경 영향 범위 판단: platform 스펙 변경은 전 서비스 영향, services 스펙 변경은 해당 서비스만 영향
- 하위 문서 동기화 필요 여부: 변경된 스펙을 직접 참조하는 문서는 반드시 동기화
- 기존 구현 불일치 처리: 불일치 코드가 있는 경우 수정 태스크를 `tasks/ready/`에 생성 후 별도 처리

## 워크플로우 조합

하나의 태스크가 여러 워크플로우에 걸칠 수 있다.

예: 새 API가 필요한 태스크
1. `contract-change` → API 계약 먼저 정의
2. `task-implementation` → 구현

예: 새 도메인 이벤트가 필요한 태스크
1. `spec-change` → 이벤트 스펙 추가/변경
2. `event-change` → 이벤트 계약 설계 및 발행자/소비자 구현
3. `task-implementation` → 태스크 구현

coordinator 에이전트가 이 판단을 자동으로 수행.
