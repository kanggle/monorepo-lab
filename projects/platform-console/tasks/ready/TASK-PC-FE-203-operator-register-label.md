# TASK-PC-FE-203 — 운영자 생성 액션 라벨 **"운영자 계정 생성 → 운영자 등록"**

**Status:** ready
**Area:** platform-console / console-web · **Scope:** 운영자 관리(/operators) 생성 액션 라벨 + IAM 가이드 네비 참조
**Type:** UI 라벨 정합 (naming — 순수 label, 도메인/동작/계약 무변화)
**Analysis model:** Opus 4.8 · **Impl 권장:** Opus/Sonnet (단순 라벨)

## Goal

운영자 생성 액션은 **로그인 계정(자격증명)을 만드는 게 아니라**, 이미 통합 IAM 계정을 가진 사람을 **운영자로
등록(admin_operators row 생성 + 역할 + 홈 테넌트)** 하는 작업이다 — 즉 **권한 주체 등록**에 가깝다. 그런데 현재
라벨 "운영자 **계정** 생성"은 "로그인 계정 발급"을 암시해(두-평면 모델과 어긋남) 혼선을 준다. (PC-FE-171이
"운영자 생성 → 운영자 계정 생성"으로 바꾼 근거 "자격증명 가진 계정을 프로비저닝"이 PC-FE-179 조사로 **틀렸음**이
드러남 — 실제로 자격증명은 회원가입이 만들고, 이 액션은 안 만든다.)

라벨을 **"운영자 계정 생성" → "운영자 등록"** 으로 바꿔 (a) 성격(권한 주체 등록)을 정확히 전달하고, (b) *역할 변경*·
*테넌트 배정* 등 기존 운영자-대상 액션과 깔끔히 구분한다("역할 부여"로 부르면 배정/역할편집과 겹침 → "등록"이 최적).

## Edge Cases

- "계정" 제거로 `/accounts`("계정 운영", 고객 계정)와의 용어 혼선도 함께 감소(운영자 영역엔 이제 "계정" 미등장).
- aria-label 변경이 접근성 쿼리 기반 테스트를 깨지 않는지 — 사전 grep 결과 테스트는 testid 기반, 라벨 단언 0건.

## Failure Scenarios

- 라벨 텍스트를 단언하는 테스트가 있었다면 RED → 사전 grep(`운영자 계정 생성`)에서 tests 매치 0건 확인(회귀 없음).
- `next lint`/`tsc`가 문자열 변경에 영향받지 않음(런타임/타입 무관) — 그래도 3종 검증으로 확증.
