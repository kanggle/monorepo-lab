# console-web — Architecture

This document declares the internal architecture of `console-web`.
All implementation tasks targeting this service must follow this declaration
and `platform/architecture-decision-rule.md`.

> **Status: v1 skeleton (ADR-MONO-013 Phase 1, TASK-MONO-108).** Boot-able minimal shell only. The real shell (IAM OIDC SSO, data-driven catalog, tenant switcher, federated domain screens) is implemented by `TASK-PC-FE-001` (Phase 1→2) and later domain-section tasks. This declaration is the target architecture all those tasks follow.
>

Retirement itself is a separate IAM project-internal spec-first task (out of scope here).

This is the **second non-IAM** binding and **reuses the § 2.4.5 per-domain credential rule verbatim** (it does not re-derive it): scm = the **IAM OIDC access token directly** (`getAccessToken()`, RS256/ADR-001, `tenant_id ∈ {scm,*}` JWT claim), **never** `getOperatorToken()`. Every inventory-visibility view **MUST** surface the producer `meta.warning: "Not for procurement decisions (S5)"` (a contract obligation). The scm-side spec-first basis is the merged scm `gateway-public-routes.md` § *platform-console operator read consumer* (TASK-SCM-BE-015). § 3 IAM-parity matrix is **not** mutated (scm is additive domain scope, not a parity row). **ADR-MONO-013 Phase 4 is COMPLETE (FE-007 wms + FE-008 scm)**; finance/erp (Phase 5/6) inherit this proven non-IAM contract.

> **Phase 5 — finance federation, ADR-MONO-013 Phase 5 COMPLETE (FE-009).** `features/finance-ops` (TASK-PC-FE-009, [`console-integration-contract.md` § 2.4.7](../../contracts/console-integration-contract.md)) binds the finance `account-service`'s existing **read-only** account + balances + transactions surface (`GET /accounts/{id}` · `/balances` · `/transactions`). This is the **third non-IAM** binding and **reuses the § 2.4.5 per-domain credential rule verbatim** (it does not re-derive it): finance = the **IAM OIDC access token directly** (`getAccessToken()`, RS256/ADR-001, `tenant_id ∈ {finance,*}` JWT claim), **never** `getOperatorToken()`. It is **strictly read-only** (finance v1 has no `admin-service` — no operator-mutation parity; finance write surface `POST /accounts` / `/kyc/upgrade` / `/holds` / `/transfers` and v2 `admin-service` explicitly out of scope): no `Idempotency-Key`/`X-Operator-Reason`/confirm/write scaffolding at all. **Honest read-surface constraint**: finance v1 has no account list/search GET — account-id-driven (no fabricated list endpoint). finance's error envelope is **flat** `{ code, message, details?, timestamp }` (same flat shape as scm but a DISTINCT producer — own parser, finance error-code vocabulary). **finance has NO documented 429** (`account-api.md` error table carries none — confirmed honestly; no fabricated backoff clause). **fintech producer obligations (finance analog of scm's S5 obligation, normative)**: F5 money = `{ amount: "<string-integer-minor-units>", currency }` — precision-exact minor-units **string**, scale-correct display via `formatMoney`, never coerced to float/JS `Number` (no `Number()`/`parseFloat`/`parseInt` on `amount` anywhere); confidential/F7 (no balance/txn/account-ref/token logging); honest regulated-state surfacing (FROZEN/RESTRICTED/CLOSED accounts + FAILED/REVERSED txns rendered, unknown enums degrade to a generic label). The finance-side spec-first basis is the merged finance `iam-integration.md` § *platform-console Operator Read Consumer* (TASK-FIN-BE-005). § 3 IAM-parity matrix is **not** mutated (finance is additive domain scope, not a parity row — § 3.1 attestation-marker count stays exactly 16). **ADR-MONO-013 Phase 5 is COMPLETE (FE-009 finance)**; erp (Phase 6) inherits the proven non-IAM contract.
>
> **finance ledger surface — second finance-product service (FE-072).** `features/ledger-ops` (TASK-PC-FE-072, [`console-integration-contract.md` § 2.4.7.1](../../contracts/console-integration-contract.md)) binds the finance **`ledger-service`**'s existing **read-only** double-entry general-ledger surface — trial balance · accounting periods (list/detail + close snapshot) · journal-entry-by-id · reconciliation discrepancy queue (`GET /api/finance/ledger/trial-balance` · `/periods[/{id}]` · `/entries/{id}` · `/reconciliation/discrepancies[/{id}]`) — making the **eleven ledger increments** (FIN-BE-007…017: posting, period close, manual posting, reconciliation, multi-currency journals, FX revaluation, FX settlement, multi-currency reconciliation) operator-visible for the first time. This is the **second service of the `finance` product** (alongside § 2.4.7 `account-service`), exactly as wms binds `admin-service` (§ 2.4.5) **and** `outbound-service` (§ 2.4.5.1). It **reuses the § 2.4.7 credential + eligibility verbatim**: domain-facing **IAM OIDC access token** (`getDomainFacingToken()`, `tenant_id ∈ {finance,*}` JWT claim), **never** `getOperatorToken()`; the ledger sits behind the same `finance.local` gateway on a distinct `/api/finance/ledger/**` path (new env `LEDGER_BASE_URL`/`LEDGER_TIMEOUT_MS`); same finance eligibility (`productKey=finance`). **Read + ONE mutation (amended by FE-073)** — FE-072 shipped strictly read-only; **TASK-PC-FE-073** carves out exactly one operator mutation, the reconciliation **discrepancy resolve** (`POST /reconciliation/discrepancies/{id}/resolve`, body `{resolutionType ∈ MATCHED_MANUALLY/WRITTEN_OFF/ACCEPTED, note}`, confirm-gated, **note(reason)-required**, **NO `Idempotency-Key`** — the producer defines none for resolve, the `409 RECONCILIATION_ALREADY_RESOLVED` state guard is the double-submit defence — **NO `X-Operator-Reason`** header, reason rides in body `note`, mirroring the erp delegation-revoke shape). Every **other** ledger mutation (`POST /entries` · `/revaluations` · `/settlements` · reconciliation ingest) stays out of scope. **Account-level drill (read, amended by FE-074)** — **TASK-PC-FE-074** surfaces the forward-declared read pair: `GET /accounts/{ledgerAccountCode}/balance` (running balance: type/normalSide/debit/credit/balance/balanceSide) + `GET /accounts/{ledgerAccountCode}/entries` (paginated lines posted to one account) — a trial-balance row's `ledgerAccountCode` is the drill key (id-driven; colon-form `CUSTOMER_WALLET:{id}` **URL-encoded** on the path), `404 LEDGER_ACCOUNT_NOT_FOUND` → inline. Read-only — adds **no** mutation (resolve stays the only one). **Reconciliation statement-detail (read, amended by FE-075)** — **TASK-PC-FE-075** surfaces the last forward-declared read: `GET /reconciliation/statements/{id}` (statement header + `matches[]` {`statementLineExternalRef`, `journalEntryId`, `money`} + `discrepancies[]` = the § 1 ingest data shape) — id-driven (no statement list/search GET; `{id}` URL-encoded), `404 RECONCILIATION_STATEMENT_NOT_FOUND` → inline; the statement view is the reconciliation **hub** (a match `journalEntryId` drills to the entry view, a discrepancy drills to the FE-073 resolve detail), wired into the existing 대사 tab (no new tab). Read-only — adds no mutation. **With FE-075 every ledger producer read is surfaced (forward-declared read 0).** **FX position open-lots drill (read, amended by FE-091)** — **TASK-PC-FE-091** surfaces the new producer read `GET /settlements/{ledgerAccountCode}/{currency}/lots` (`ledger-api.md` § 12, FIN-BE-028, the 20th increment, added after FE-075): the open FX acquisition lots for one `(account, currency)` position — `lots[]` {`lotId`, `currency`, `acquiredAt`, `seq`, `originalForeignMinor`, `remainingForeignMinor`, `originalBaseMinor`, `carryingBaseMinor`, `sourceJournalEntryId`} + summary {`totalRemainingForeignMinor`, `totalCarryingBaseMinor`, `lotCount`} — making the ADR-001 FIFO/lot accounting (17th incr consumption + 18th incr revaluation mark-to-spot) operator-visible in a new "FX 포지션 로트" tab. id-driven by the `(ledgerAccountCode, currency)` pair (colon-form code **URL-encoded**; currency a 3-letter ISO-4217 code); an **empty position** is the producer's `200` empty-state (`lots: []` / totals `"0"` / `lotCount 0`) → an empty-state message (never a 404), an unsupported currency → `400 VALIDATION_ERROR` → inline. Every `*Minor` field is an F5 minor-units **string** (rendered via `formatMoney` from the string, never coerced to a number; `seq`/`lotCount` are plain numbers). Read-only — adds **no** mutation (resolve stays the only one). **F5 money (multi-currency ledger form)**: each journal line carries `money` + `exchangeRate` + `baseAmount` (KRW, balance-authoritative) — all rendered precision-exact from the minor-units **string** (never coerced to float/`Number`; tests grep-assert). Honest state surfacing (sourceType TRANSACTION/MANUAL/REVALUATION/SETTLEMENT · period OPEN/CLOSED · discrepancy AMOUNT_MISMATCH/OPEN/RESOLVED — incl. the 11th-increment FX-difference matched pair). flat finance error envelope; no documented 429; confidential/F7 (no balance/line/code/token logging). § 3 IAM-parity matrix is **not** mutated (additive — § 3.1 marker count stays 16).

## Architecture Style

**Layered by Feature** — 기능(도메인 섹션) 단위 폴더로 묶고 `shared/` 공통 레이어만 분리한다. Feature-Sliced Design 의 엄격한 5계층 분리는 본 도메인 규모에 비해 오버엔지니어링이라 채택하지 않는다.

참조 skill: [.claude/skills/frontend/architecture/layered-by-feature/SKILL.md](../../../../../.claude/skills/frontend/architecture/layered-by-feature/SKILL.md)

## Why This Architecture

- **운영자 단일 사용자 그룹** — 모든 화면이 동일 사용자(플랫폼 운영자)를 대상으로 한다. IAM `admin-web` 가 동일 trade-off 로 Layered by Feature 를 택했고, `console-web` 은 그 운영자 표면을 흡수(ADR-MONO-013 D4)하므로 동일 스타일 유지가 cross-stack 리뷰 학습 비용을 최소화한다.
- **도메인 섹션 = feature** — iam/wms/scm 각 도메인 운영 화면이 자연스러운 feature 경계다. 섹션 간 강한 격리(FSD `entities/`+`widgets/`)의 이득보다 ceremony 비용이 크다. 도메인 추가(erp/finance)는 feature 폴더 + 카탈로그 레지스트리 엔트리 추가로 끝난다.
- **list/detail/action CRUD 반복** — 계정 검색→상세→액션, 감사 조회 등 admin-web parity 표면이 list/detail 패턴 반복. admin-dashboard / admin-web 의 선택과 일치.
- **Server-component 우선** — 데이터 페칭은 서버에서. client features 격리의 가치는 leaf-level client component 에 한정.

## Styling & Theming

Vercel-style design-token system (TASK-PC-FE-038). This is the **canonical**
theming for the operator suite; `web-store` mirrors the same "near-white primary
in dark" approach with its own plain-CSS token set.

- **Design tokens** — semantic colours are CSS custom properties on `:root` in `app/globals.css` (Geist neutrals as HSL channel triples): `--background` / `--foreground` / `--border` / `--muted(-foreground)` / `--primary(-foreground)` / `--accent(-foreground)` / `--destructive(-foreground)` / `--ring`. `tailwind.config.ts` maps its colour scale to `hsl(var(--token))`, so components use semantic classes (`bg-background` / `text-foreground` / `border-border` …) and the whole palette rethemes from one token set.
- **Light / dark** — `tailwind.config.ts` `darkMode: 'class'`; `globals.css` `.dark` flips the token palette. **next-themes** (`shared/ui/ThemeProvider`, wrapped in the root `app/layout.tsx`) drives the `.dark` class on `<html>` with `attribute="class"`, `defaultTheme="system"`, `enableSystem`, `disableTransitionOnChange`, and injects a pre-hydration script so the correct theme paints with no flash (root `<html suppressHydrationWarning>`). `shared/ui/ThemeToggle` (top bar, `(console)/layout.tsx`) flips light/dark and renders a mount-guarded sun/moon icon to avoid an SSR/client hydration mismatch.
- **Shared table style** — the `.data-table` component class (`globals.css`, TASK-PC-FE-042) gives every console table the rounded, bordered Vercel look from one class: `border-separate` + `border-spacing-0` + `overflow-hidden` clips the rounded outer frame (the collapsed-border model ignores `border-radius`), and horizontal dividers move to cell `border-b` since the separate-borders model ignores per-`<tr>` borders.

## Allowed Dependencies

| Layer | 허용 import |
|---|---|
| `app/` | `features/*`, `shared/*` |
| `features/<F>/` | 자체 폴더 내부, `shared/*` |
| `shared/` | 자체만 (features / app 금지) |

의존성은 **위에서 아래로만** 흐른다 (`app/ → features/ → shared/`). 같은 계층 `features/A → features/B` 상호 참조 금지(공유 가치는 `shared/` 로 승격).

## Forbidden Dependencies

- `features/A` → `features/B` 상호 참조
- 컴포넌트에서 `fetch()` 직접 호출 — 모든 백엔드 호출은 `shared/api/` 클라이언트 경유
- JWT / refresh token 을 `localStorage` / `sessionStorage` 에 저장 (HttpOnly 쿠키만 — `frontend-app.md` 위반)
- 빌드 타임 시크릿 주입 — 런타임 env 만
- 다른 앱(`admin-web`, `web-store`, …)의 `features/` 직접 import — 공유는 `@repo/` 레벨만

## Boundary Rules

- `app/` 라우트는 비즈니스 로직 없이 `features/*`+`shared/*` 합성에 한정.
- 도메인 섹션(`features/<domain>-ops/`)은 서로 직접 참조하지 않는다 — 공유 가치는 `shared/` 승격.
- `shared/api/` 는 **유일한** 백엔드 진입점 — cookie 자동 전달 + 401 refresh + per-domain 테넌트 헤더(`X-Tenant-Id`) 부착이 여기서 통과.
- 클라이언트 권한 가드는 UX 게이트일 뿐 — 실제 권한·격리는 도메인 백엔드(IAM/gateway)가 최종 책임 (cross-tenant 거부 포함).

## Server vs Client Components

- **Default: Server Component** — 데이터 페칭은 서버에서.
- **`'use client'`**: 폼·다이얼로그·스위처 등 상호작용 리프만.
- **React Query** 는 client component 한정; 서버는 `fetch` + `revalidate`/`cache`.

## Auth Flow

콘솔은 IAM OIDC **public client (`platform-console-web`, Authorization Code + PKCE)**. (Phase 2 / TASK-PC-FE-001)

1. `/login` → IAM `/oauth2/authorize` (PKCE `code_challenge`) 리다이렉트
2. IAM 콜백 → `/api/auth/callback` (Next.js route handler) 가 `code`+`code_verifier` 로 `/oauth2/token` 교환
3. 성공 시 `Set-Cookie: accessToken/refreshToken; HttpOnly; Secure; SameSite=Strict`
4. **서버측 operator-token 교환** (ADR-MONO-014 D2 / TASK-PC-FE-002a) — IAM access token 을 IAM `admin-service POST /api/admin/auth/token-exchange` (RFC 8693, subject_token=IAM access token) 으로 단명 operator token (`token_type=admin`, `iss=admin-service`) 으로 교환해 별도 HttpOnly operator 쿠키에 저장. IAM OIDC token 은 `/api/admin/**` 자격이 **아니며** subject token 입력으로만 사용된다. exchange `401`(미프로비저닝) → 강제 재로그인(`not_provisioned`); timeout/unreachable/5xx → 세션 불가(operator 쿠키 미설정, partial authed state 금지 — IAM token 으로 fallback 안 함).
5. 이후 모든 요청 쿠키 자동 전달; 401 → `/api/auth/refresh` → IAM 쿠키 회전 + **operator-token 재교환**(re-exchange 모델, operator-refresh state 없음) → 재시도
6. 로그아웃 → IAM revoke + IAM/operator 쿠키 삭제

IAM access token·operator token 모두 **절대 JavaScript 접근 불가** (HttpOnly) — `frontend-app.md` 필수. 교환 계약 상세는 IAM [`admin-api.md` §`POST /api/admin/auth/token-exchange`](../../../../iam-platform/specs/contracts/http/admin-api.md) + [`admin-service/security.md` §IAM OIDC Subject-Token Validation](../../../../iam-platform/specs/services/admin-service/security.md), 소비 의무는 [`console-integration-contract.md` § 2.1/§ 2.6](../../contracts/console-integration-contract.md).

### Per-domain credential selection (정규 규칙 — `console-integration-contract.md` § 2.4.5)

콘솔이 도메인 `/api/admin/**` 표면에 제시하는 자격은 **도메인별로 다르며**, 각 § 2.4.x 바인딩이 자기 자격을 명시한다. 한 도메인의 인증 모델을 다른 도메인에 일괄 적용하면 안 된다 (보안 결함).

| 도메인 바인딩 | `/api/admin/**` 자격 | 메커니즘 | 권위 |
|---|---|---|---|
| IAM (§§ 2.4.1–2.4.4) | **exchanged operator token** (`getOperatorToken()`) | server-side RFC 8693 교환 (§ 2.6) | ADR-MONO-014; **#569 trust-boundary invariant** — IAM OIDC token 은 IAM `/api/admin/**` 에 **결코** 도달 안 함 |
| **wms (§ 2.4.5, TASK-PC-FE-007)** | **IAM OIDC access token** (`getAccessToken()`, FE-001 IAM-session 쿠키) | `Authorization: Bearer <IAM OIDC access token>` **직접** | wms `admin-service-api.md` § Global Conventions + `iam-integration.md`: IAM RS256 JWT (ADR-001), JWKS 검증, **`tenant_id=wms` 는 JWT claim 에서 producer-side 강제**. wms 는 token-exchange **없음** — IAM OIDC token 을 **요구** |

**#569 invariant 은 IAM-domain-scoped 이며 wms 로 일반화되지 않는다**: #569 는 IAM OIDC token 을 **IAM** `/api/admin/**` 에 금지한다 — IAM 가 거기서 § 2.6 exchanged token 을 요구하기 때문. wms 게이트웨이는 정반대로 IAM OIDC token 을 **요구**한다 — 둘은 충돌이 아니라 서로 다른 per-domain 바인딩이다. 구현자는 (a) IAM operator-token-exchange (§ 2.6) 를 wms 로 옮겨서도, (b) "admin path 의 IAM token" 을 보편적 #569 위반으로 취급해서도 안 된다. `features/wms-ops` 클라이언트는 도메인-facing 토큰을 쓰고 `getOperatorToken()` 은 **절대** 쓰지 않는다 (테스트가 FE-002..006 단언의 역으로 고정). finance/erp (Phase 5/6) 콘솔 섹션은 이 명시된 규칙을 상속한다 — 추측이 아니라 각 producer 인증 계약에 대해 자격을 명시 선언한다.

**wms 두 번째 surface — `features/wms-outbound-ops` (TASK-PC-FE-057, § 2.4.5.1)**: wms `outbound-service` (pick→pack→ship operator leg) 는 §2.4.5 의 **같은 domain-facing IAM-OIDC 자격**(`getDomainFacingToken()`, 절대 operator token 아님) + **같은 게이트웨이·JWT `tenant_id=wms` claim**(`X-Tenant-Id` 미전송) + **같은 nested error envelope** 을 상속하고, **다른 base URL** `WMS_OUTBOUND_BASE_URL`(기본 `http://wms.local/api/v1/outbound` — §2.4.5 `WMS_ADMIN_BASE_URL` 의 `/api/v1/admin` 과 **distinct path prefix**, 같은 게이트웨이 hostname)만 더한다. compound mutation(Pick=list→confirm-as-planned / Pack=create→seal) 은 proxy route 에서 server-side orchestrate 하며 각 producer call 이 자기 idempotency key 를 갖고, optimistic-lock `409 CONFLICT` 는 refetch+retry-prompt(silent auto-retry 금지).

**scm 두 번째 surface — `features/scm-replenishment` (TASK-PC-FE-077, § 2.4.6.1)**: scm `demand-planning-service` (보충 추천 approve/dismiss operator gate) 는 §2.4.6 의 **같은 domain-facing IAM-OIDC 자격**(`getDomainFacingToken()`, 절대 operator token 아님 — scm 은 token-exchange 없음) + **같은 게이트웨이·JWT `tenant_id ∈ {scm,*}` claim**(`X-Tenant-Id` 미전송) + **같은 flat error envelope** + **같은 429 Retry-After bounded backoff** 를 상속하고, **같은 base URL** `SCM_GATEWAY_BASE_URL`(기본 `http://scm.local`)의 `/api/v1/demand-planning/**` path 만 더한다. **첫 scm operator-mutation**: approve/dismiss 는 POST + optional body(`{note}`/`{reason}`)이며 **no `Idempotency-Key`**(producer 미정의·server-side idempotent by suggestion state) · **no `X-Operator-Reason`**(reason=body) · confirm-gated. approve = DRAFT PO materialise only(ADR-MONO-027 D5 human-gate) — 이 화면은 PO submit/confirm/cancel 을 발행하지 않는다.

### 도메인 랜딩 운영 개요 스냅샷 — 공통 read-leg (TASK-PC-FE-168 결정 · gating design)

각 도메인 섹션 랜딩(`/wms` `/scm` `/finance` `/erp` `/ecommerce`)을 **운영 개요 스냅샷**(per-area 카운트 + 핵심 상태 분포 + 최근 활동)으로 승격하는 시리즈(PC-FE-156 ecommerce=direct 레퍼런스, PC-FE-166 wms=bff-도메인 레퍼런스, 167/160/161 후속, 162 capstone)의 **공통 read-leg 방식을 여기서 한 번에 확정**한다 (4개 도메인 개별 결정 시 divergence 위험).

**실측 정정 (프레이밍 오류 교정)**: wms/scm/finance/erp 섹션 페이지는 이미 **console-web DIRECT**(`getDomainFacingToken()` → 각 도메인 gateway/admin base URL, §§ 2.4.5–2.4.8)로 producer 를 호출한다 — ecommerce(§ 2.4.10)와 **동일 모델**이다. "console-bff READ-leg(§ 2.4.9.1/.2)"는 **콘솔 홈** cross-domain 대시보드(Operator Overview / Domain Health)만 해당하고, 도메인 섹션 랜딩이 아니다.

**결정 — console-web DIRECT fan-out 재사용 (Option 1)**: 4개 bff-도메인 개요 스냅샷은 ecommerce(§ 2.4.10.6)와 동일한 console-web DIRECT fan-out을 쓴다. 새 console-bff leg 신설·producer 변경 없음.

- **Read model**: 각 도메인의 **기존 소비 중인** `list*` 서버 함수를 재사용해 server-side fan-out. credential = domain-facing IAM OIDC token(`getDomainFacingToken()`, `X-Tenant-Id` 미전송 — JWT `tenant_id` claim; 위 per-domain 규칙 § 2.4.5 상속).
- **카운트 = `totalElements`**: 각 area 카운트 = 기존 list endpoint 의 `totalElements` 를 `?page=0&size=1` 로 읽음. **ADR-MONO-017 D3.B** — producer `/summary`/집계 endpoint **금지**, producer retrofit **금지**. (ecommerce 는 §2.4.10 console-absorption 도메인으로서 나중에 전용 `/summary`(BE-468/PC-FE-164)를 얻었으나, **비-흡수 federation 인 bff-도메인들은 그 treatment 를 받지 않는다** — 그것이 D3.B 가 금지하는 producer retrofit 이기 때문. bff-도메인은 `totalElements` 파생만.)
- **왜 console-bff leg 가 아닌가**: BFF(§ 2.4.9)는 cross-domain 합성(콘솔 홈)을 위한 것 — 단일 도메인 내부 스냅샷은 server-side fan-in 이 불필요하고, DIRECT 클라이언트가 이미 per-domain credential + § 2.5 resilience taxonomy 를 갖춘다. bff leg 신설은 새 route+계약+백엔드 작업을 무이득으로 추가.
- **Resilience (§ 2.5)**: per-cell degrade 는 cell-local(한 area 실패가 스냅샷을 blank 하지 않음); **어느 leg 든 `401` → whole-session `redirect('/login')`**(partial authed state 금지 — ecommerce `cell()` 미러). read-only, auto-refetch 없음.
- **per-domain 편차**:
  - **wms (PC-FE-166, 레퍼런스)**: 재고/배송 `totalElements` 카운트 타일(배송=오늘/주간/월간/전체 기간, PC-FE-174) + 알림 확인상태 분포(미확인/확인) + 최근 출고. 알림은 규모-타일이 아니라 확인상태 분포로만 표현(총계 타일은 분포 합과 중복 → PC-FE-170 제거). 카운트 타일은 **비-링크 stat 타일**. 조회 테이블은 개요에서 분리되어 전용/기존 라우트로 이동(재고→`/wms/inventory` PC-FE-173, 택배/출고→`/wms/outbound` PC-FE-175); 개요는 스냅샷 타일 + 최근 출고 glance + 알림 테이블만 유지.
  - **scm (PC-FE-167)**: PO `totalElements` + PO-상태 분포 + 최근 PO. S5 `meta.warning` freshness honesty 유지. 429 는 client 가 이미 bounded backoff(§ 2.4.6) — 개요는 re-storm 안 함.
  - **finance (PC-FE-160, PARKED) → TASK-PC-FE-229 재개**: finance v1 은 계좌 list/search GET **없음**(account-id-driven, § 2.4.7) — **계좌 카운트 fan-out 은 여전히 불가, 합성 ₩ 집계도 여전히 금지**(이 제약은 완화되지 않음). 그러나 **원장(ledger-service)은 이미 browsable index read**(시산표·기간·대사·FX, § 2.4.7.1)를 제공하고, **운영자 본인의 기본계좌 단건**(`getFinanceDefaultAccountId`→`getAccount`/`getBalances`, 목록 아님)은 별도 producer read 없이 조회 가능 — 이 두 가지만으로 정직한(non-fabricated) 개요가 성립함을 확인, `/finance` 도메인 루트를 IAM/WMS/SCM/E-Commerce 와 동일하게 개요 랜딩으로 전환(§ 2.4.7.2, `features/finance-overview`). account-lookup 표면은 `/finance/accounts` 로 이동(라벨 `운영`→`계좌`).
  - **erp (PC-FE-161 → TASK-PC-FE-232 승격)**: 5 master + 결재 대기(approval inbox) + 활성 위임(delegation ACTIVE) `totalElements` — 개요는 독립 `/erp` 랜딩(구 마스터-임베드에서 승격, 마스터는 `/erp/masters` 로 이동). thin surface(7 count, 분포/최근 없음).
- **§ 3 parity**: 이미 열거된 endpoint 만 새 read composition 으로 소비 — 새 § 3 row 없음, count 불변.

**wms(PC-FE-166)가 bff-도메인 패턴의 첫 레퍼런스 구현**(direct-model 의 ecommerce PC-FE-156 대응)임을 확정한다.

### Active-tenant 스위처 → assume-tenant 교환 (ADR-MONO-020 D4 / `console-integration-contract.md` § 2.7)

콘솔은 **두 개의 server-side RFC 8693 교환**을 가진다 (혼동 금지):

1. **operator-identity 교환** (ADR-MONO-014, § 2.6): admin **JSON** `POST /api/admin/auth/token-exchange` → `/api/admin/**` operator token. 변경 없음.
2. **assume-tenant 교환** (ADR-MONO-020 D4, § 2.7, **신규**): SAS **form-urlencoded** `POST ${OIDC_ISSUER_URL}/oauth2/token`, `grant_type=...token-exchange` + `subject_token=<base IAM OIDC access token>` + `audience=<선택 tenant>`. → 선택된 customer 로 re-scope 된 단명 domain-facing IAM OIDC token (`tenant_id=<선택>` + `entitled_domains=<선택의 ACTIVE subs>`). **refresh_token 없음** — selection / IAM refresh 마다 재발급. producer 계약은 IAM [`auth-api.md § Assume-Tenant Exchange`](../../../../iam-platform/specs/contracts/http/auth-api.md) (TASK-BE-327) 소유 — 소비만, 와이어 shape 재정의 금지.

**assumed-token 세션 + domain-facing resolver**: `shared/lib/session.ts` 에 `ASSUMED_TOKEN_COOKIE`(`console_assumed_token`, HttpOnly·Secure·Lax) + `getAssumedToken()` + **`getDomainFacingToken()` = `getAssumedToken() ?? getAccessToken()`** (중심 resolver). 모든 tenant-scoped 도메인 read 의 IAM-OIDC bearer = `getDomainFacingToken()`:

- cross-domain overview 프록시(§ 2.4.9.1) `Authorization: Bearer` (BFF 의 non-IAM fan-out leg 이 verbatim forward; IAM leg 은 `X-Operator-Token` 유지 — § 2.6 불변);
- 4 non-IAM 도메인 클라이언트(`features/{wms,scm,finance,erp}-ops`, §§ 2.4.5–2.4.8) — per-domain credential 규칙 § 2.4.5 불변, **어느 IAM OIDC token** 인지만 변경.
- **IAM 도메인 클라이언트**(`features/{accounts,audit,operators,dashboards}` → `getOperatorToken()`)는 **변경 없음** — operator-token 경계(§ 2.1/§ 2.6, #569)는 그대로. `getDomainFacingToken()` 은 결코 IAM `/api/admin/**` 자격이 아니다.
- **net-zero**: 미전환 / single-tenant 운영자는 assumed token 이 없어 `getDomainFacingToken()` = base token → 기존 동작 byte-identical.

**스위치 라우트(`POST /api/tenant`)**: 기존 registry-membership allow-check (defence-in-depth — 유지) 후 assume-tenant 교환 호출; 성공 시 `console_assumed_token`(maxAge=`expires_in`) + `console_active_tenant=<tenant>` 를 **atomically** 설정. assumed token 은 현재 active tenant 로 construction-scoped. **fail-closed**: `denied`(producer `400 invalid_grant` — D2 assignment 게이트/subject 무효/admin-service 장애) → 403, **쿠키 무변경**(이전 선택 보존); `invalid`(`400 invalid_request`) → 422; `unavailable`(5xx/timeout/network) → 503; base token 부재 → 401. base IAM token 은 `subject_token` 입력일 뿐 — **미로깅·미반환**; 선택-tenant 경계에서 base token 으로 **절대 fallback 안 함**. `tenant=''` clear → **두 쿠키 모두** 삭제(coupled).

**refresh 재-assume**: assumed token 은 refresh token 이 없으므로 IAM refresh 시 operator-token 재교환 후 active tenant 가 있으면 rotated base token 으로 **재-assume**. 재-assume 실패 → assumed token + active tenant 모두 drop (base/no-tenant 로 fallback — stale assumed token 금지); base IAM+operator 세션은 유효. 전체 세션 drop(IAM refresh 거부 / operator 재교환 fail-closed) 시에도 assumed token + active tenant clear.

**선택 가능 tenant**: 스위처의 선택 가능 집합 = ConsoleRegistry effective scope (TASK-BE-326 dual-read: assignment rows ∪ legacy home tenant) — multi-assignment 운영자는 할당된 모든 customer 가 노출 (BE-326 가 이미 wire — 콘솔 변경 불필요). D4 는 selection 시 credential re-scope 만.

## Integration Rules

`console-web` 은 `frontend-app` 으로서 **백엔드 컨트랙트의 소비자**이며 자체 컨트랙트·영속 상태를 소유하지 않는다.

- **통합 계약**: [specs/contracts/console-integration-contract.md](../../contracts/console-integration-contract.md) — OIDC public client / product·tenant 레지스트리 / per-domain console-facing API / resilience. 콘솔은 envelope 를 소유하지 않으며, 도메인 spec 선행 변경 후 follow.
- **인증/세션**: IAM `auth-service` (OIDC AS, ADR-001) — public client Auth Code+PKCE. IAM-side OIDC client + 레지스트리 surface 는 선행 task `TASK-BE-296` (IAM project-internal). `/api/admin/**` operator 자격은 IAM `admin-service POST /api/admin/auth/token-exchange` (RFC 8693, ADR-MONO-014) 으로 server-side 교환한 operator token — IAM OIDC token 은 subject token 입력으로만 쓰이며 결코 `/api/admin/**` 자격이 아니다 (선행 task `TASK-BE-298`, IAM project-internal, merged).
- **도메인 호출**: 각 도메인 `gateway`/`admin` REST API (server-side, 테넌트 스코프). 도메인별 endpoint 스키마는 그 도메인 `specs/contracts/` 소유, 섹션 빌드 시 cross-ref.
  - **IAM accounts (`features/accounts`, TASK-PC-FE-002)**: 8 operator 작업(search/detail/lock/unlock/bulk-lock/revoke-session/gdpr-delete/export)을 server-side 로 호출. authoritative producer 계약 = IAM [`admin-api.md`](../../../../iam-platform/specs/contracts/http/admin-api.md) (`GET /api/admin/accounts`, `POST /api/admin/accounts/{id}/lock`, `POST /api/admin/accounts/bulk-lock`, `POST /api/admin/accounts/{id}/unlock`, `POST /api/admin/sessions/{accountId}/revoke`, `POST /api/admin/accounts/{id}/gdpr-delete`, `GET /api/admin/accounts/{id}/export`) — **변경 없음, 소비만**. 소비 의무(operator token / `X-Tenant-Id` / mutation `X-Operator-Reason`+`Idempotency-Key` / resilience)는 [`console-integration-contract.md` § 2.4](../../contracts/console-integration-contract.md) 의 IAM-accounts cross-reference 가 canonical.

## Change Rule

`console-web` 은 컨트랙트·도메인 로직·영속 상태를 소유하지 않으므로 본 Change Rule 은 **UI·소비 컨트랙트 호환성·번들 예산**을 규정한다.

1. 소비하는 통합 계약([console-integration-contract.md](../../contracts/console-integration-contract.md)) / 도메인 컨트랙트 변경은 **소유 측 spec 선행** — console-web 은 그 후 `shared/api/*` 타입+파서+contract test 동시 갱신 follow.
2. 라우트·feature 폴더 구조 변경은 § Internal Structure Rule + § Allowed Dependencies 정합 유지 (Layered by Feature 위반 차단).
3. 인증 흐름(OIDC public client / PKCE / HttpOnly 쿠키 / refresh route / **operator-token 교환** ADR-MONO-014) 변경은 [frontend-app.md](../../../../../platform/service-types/frontend-app.md) 필수 요구 + IAM `auth-service`·`admin-service` 호환성 확인 선행. IAM access token·operator token JS 접근 불가 + IAM OIDC token 이 `/api/admin/**` 자격이 아니라는 불변식 유지.
4. § Performance Budget 회귀 동반 변경 차단 (CI 강제).
5. 도메인 동작 변경 필요 시 소유 도메인 task 로 분리 — console-web 은 호출·표현만. IAM `admin-web` 폐기는 ADR-MONO-013 Phase 3 (parity-gated, IAM project-internal) — console-web task 아님.
